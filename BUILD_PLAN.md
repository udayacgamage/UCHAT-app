### Step 5: Encryption Service
**Complexity**: High | **Duration**: 2.5 hours | **Blocker**: Critical for security

**What to do**:
Implement AES-256 message encryption using `encrypt` package:

1. **Key Derivation**
   - Derive unique key per chat from both user IDs (deterministic so both users get same key)
   - Example: `hash(userId1 + userId2)` → AES key
   - Store key in `flutter_secure_storage` under key: `chat_{chatId}_key`

2. **Encryption Methods**
   ```dart
   String encryptMessage(String plainText, String chatId)
     → Returns base64-encoded cipher text
   
   String decryptMessage(String cipherText, String chatId)
     → Returns plain text
   
   Future<void> generateAndStoreChatKey(String chatId)
     → Derives key and stores securely
   
   Future<String?> getChatKey(String chatId)
     → Retrieves key from secure storage
   ```

3. **Requirements**
   - Use AES-256-CBC mode
   - Generate random IV for each message
   - Store IV with cipher text (prepend to output)
   - Padding: PKCS7 (automatic with `encrypt` package)

**Files to create**: `lib/services/encryption_service.dart`

**Critical Points**:
- Key derivation must be **identical** on both sender and receiver
- IV must be random (don't reuse)
- Test with actual encryption/decryption round-trip
- Store encrypted messages as base64 in Firestore

**Validation**:
- ✓ Encryption and decryption round-trip successfully
- ✓ Both users derive identical keys
- ✓ Keys stored securely (verify with secure storage)
- ✓ Different messages produce different ciphers (due to random IV)

**Depends on**: Step 1 (packages), Step 3 (logger)  
**Blocks**: Chat service (Step 8), chat screen (Step 18)

   rules_version = '2';
   service cloud.firestore {
     match /databases/{database}/documents {
       // Users can only read/write their own user document
       match /users/{uid} {
         allow read, write: if request.auth.uid == uid;
       }
       
       // Users can only read a chat if they are participants
       match /chats/{chatId} {
         allow read: if request.auth.uid in resource.data.participants;
         allow write: if request.auth.uid in resource.data.participants;
         
         // Messages
         match /messages/{messageId} {
           allow read: if request.auth.uid in get(/databases/$(database)/documents/chats/$(chatId)).data.participants;
           allow create: if request.auth.uid in get(/databases/$(database)/documents/chats/$(chatId)).data.participants
                         && request.resource.data.senderId == request.auth.uid;
           allow update: if request.auth.uid == resource.data.senderId 
                         || (request.auth.uid in get(/databases/$(database)/documents/chats/$(chatId)).data.participants);
           allow delete: if request.auth.uid == resource.data.senderId;
         }
         
         // Typing indicator
         match /typing/{userId} {
           allow write: if request.auth.uid == userId;
           allow read: if request.auth.uid in get(/databases/$(database)/documents/chats/$(chatId)).data.participants;
         }
       }
       
       // Groups
       match /groups/{groupId} {
         allow read: if request.auth.uid in resource.data.members;
         allow create: if request.auth.uid != null;
         allow update: if request.auth.uid in resource.data.adminIds;
         allow delete: if request.auth.uid in resource.data.adminIds;
         
         // Group messages (same as chat messages)
         match /messages/{messageId} {
           allow read: if request.auth.uid in get(/databases/$(database)/documents/groups/$(groupId)).data.members;
           allow create: if request.auth.uid in get(/databases/$(database)/documents/groups/$(groupId)).data.members
                         && request.resource.data.senderId == request.auth.uid;
           allow update, delete: if request.auth.uid == resource.data.senderId 
                                  || request.auth.uid in get(/databases/$(database)/documents/groups/$(groupId)).data.adminIds;
         }
       }
     }
   }

   ## Firestore Security Rules
   
   Deploy the `firestore.rules` file to your Firestore database:
   ```bash
   firebase deploy --only firestore:rules
   ```
   
   Rules enforce:
   - Users can only read/write their own user document
   - Users can only access chats they are participants of
   - Only message sender can delete their messages
   - Group admins can manage group members
   
   ## Encryption Details
   
   ### Message Encryption (AES-256)
   - Each direct chat generates a unique encryption key
   - Key derivation: `SHA-256(sortedUserIds)` → AES key
   - Each message encrypted with random IV
   - IV prepended to ciphertext, both base64-encoded
   - Decryption transparent to UI layer
   
   ### Secure Storage
   - Firebase Auth tokens stored via `flutter_secure_storage`
   - Encryption keys stored in secure storage (not Firestore)
   - Device-level encryption via platform (Android KeyStore)
   
   ## Security Notes
   
   - ✅ End-to-end encryption protects message content from Firebase
   - ✅ Biometric lock prevents unauthorized app access
   - ⚠️ Metadata (sender, recipient, timestamp) not encrypted
   - ⚠️ Firebase Admin SDKs can read encrypted content with database access
   - ⚠️ This is a reference implementation; audit before production use
   
   ## Known Limitations
   
   - Encryption key shared only via user ID derivation (no key exchange protocol)
   - Media files not encrypted (only metadata encrypted)
   - No message forwarding encryption handling
   - No perfect forward secrecy (same key throughout chat lifetime)
   - Typing indicator and online status sent unencrypted

## 🚨 Critical Integration Points

1. **Encryption ↔ Chat Service**
   - Message content MUST be encrypted before Firestore write
   - Message content MUST be decrypted after Firestore read
   - Encryption key must exist before first message

2. **Auth State ↔ Router Redirect**
   - Router guards depend on real-time auth state
   - If auth state loading, show splash screen
   - If user null, redirect to login
   - If user exists, redirect to home

3. **FCM Token ↔ Auth Service**
   - FCM token fetched during login/signup
   - Token stored in secure storage
   - Token sent to Firestore user document
   - Token refreshed periodically

4. **Notification Tap ↔ GoRouter**
   - Notification payload includes `chatId`
   - Notification tap triggers navigation via GoRouter
   - Navigation must use `context.go()` or `context.push()`

5. **Real-time Streams ↔ UI Updates**
   - Chat list updates via `chatsStreamProvider`
   - Messages update via `messagesStreamProvider`
   - Typing status updates via `typingStatusProvider`
   - Online status updates via user watch stream
   - UI must handle loading/error/data states

6. **Message Pagination**
   - Chat service loads 30 messages initially
   - On scroll-to-top, load older messages using `startAfter(lastDocument)`
   - Show loading skeleton during pagination
   - Don't duplicate messages in list

## ⚠️ Known Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Firebase project misconfiguration | Auth/Firestore/Storage fail | Complete Firebase setup and test connections before starting services |
| AES key derivation not deterministic | Users can't decrypt each other's messages | Test key derivation with multiple user pairs; use same algorithm on sender/receiver |
| Firestore security rules too restrictive | Legitimate users can't send/receive messages | Test rules with real user pairs; use Firestore emulator during development |
| FCM token expired | Push notifications fail | Implement token refresh handler; re-request token on app resume |
| Message pagination offset lost | Messages duplicate or skip | Use Firestore `startAfter(DocumentSnapshot)` consistently; test pagination thoroughly |
| Biometric not available | Lock screen crashes | Gracefully fall back to PIN; check `canCheckBiometrics` before prompting |
| Unencrypted metadata in Firestore | Metadata (sender, recipient, time) visible to Firebase | Accept limitation; encrypt metadata in Phase 2 if needed (requires additional design) |
